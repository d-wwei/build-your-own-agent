# 模块 11：横切关注点
> 三个真正跨越多个模块的关注点 -- 以及关于随意称呼事物为"横切"的警告。

## 它是什么

横切关注点是无法被干净地分配到单个模块的事物。它们同时触及多个层。

在实践中，在四个项目中，只有三件事符合条件：

1. **可观测性** -- 日志、指标、追踪。每个模块都发射数据。某个东西必须收集、结构化和路由这些数据。
2. **Plugin 框架** -- 一个让第三方代码同时注册工具、通道、上下文引擎和模型提供商的系统。不是工具注册表（那是模块 3）。是一个同时跨越多个注册表的框架。
3. **热配置重载** -- 在不重启 Agent 的情况下更改运行时配置。需要所有读取配置的模块之间的协调。

大多数人称之为"横切"的东西并不是。命令审批只是 Tools 层。记忆注入只是 Core + Memory 通过 Context Manager 实现。权限检查只是安全信封。这些感觉像横切的，是因为它们跨功能影响用户体验，但在架构上它们存在于一个模块中，有清晰的接口。

## 它解决什么问题

**可观测性解决"刚才发生了什么"。** 你的 Agent 调用了一个工具，工具失败了，模型用不同参数重试，重试成功了但花了 14 秒。没有结构化日志，你知道最终结果但不知道路径。没有指标，你不知道这种情况 30% 的时间都在发生。没有追踪，你无法将工具失败与产生畸形参数的上游模型调用联系起来。

**Plugin 框架解决"第三方如何扩展这个系统"。** 如果你构建一个供他人部署和定制的平台，他们需要在不 fork 你源码的情况下添加能力。一个只注册工具的插件系统不是横切的 -- 它只是带有扩展点的 Tools 模块。当单个插件可以在一次激活调用中同时注册工具、通道、上下文引擎和模型提供商时，插件系统才真正成为横切的。

**热配置重载解决"如何在生产环境中更改设置"。** 你的 Agent 服务 50 个并发用户。你需要将默认模型从 GPT-4 切换到 Claude。没有热重载，你重启进程并中断所有 50 个会话。有了热重载，你更新配置文件，Agent 检测到变更，新请求使用新模型，而现有会话在旧模型上完成。

## 四个项目如何实现

### 可观测性

**hermes-agent：** Python `logging` 模块。标准日志级别（DEBUG、INFO、WARNING、ERROR）。无结构化格式 -- 纯文本日志行。无指标收集。无分布式追踪。对于本地运行的单用户 Agent 足够好。当你需要调试跨多个会话的生产问题时就不够了。

**openclaw：** Sentry 集成用于错误追踪和性能监控。通过插件系统的结构化日志事件。四个项目中最面向生产的可观测性，但仍然侧重于错误情况而非全面的遥测。

**NemoClaw：** `gateway-state.ts` 中的网关健康状态分类（healthy、degraded、dead）。SSH + curl 探测用于存活检测。这是服务于基础设施层的可观测性 -- 关注"运行时是否存活"而非"Agent 在做什么"。唯一一个由可观测性信号触发自动恢复的项目。

**Claude Code：** Anthropic 使用分析的内部遥测。不对用户或运维人员暴露。从开源角度看，实际上没有可观测性。

### Plugin 框架

**hermes-agent：** 没有 Plugin 框架。工具通过 `registry.register()` 在 import 时注册。想添加工具？编辑源码。这是工具注册表，不是插件系统。对于不需要第三方可扩展性的 Agent 来说也完全足够。

**openclaw：** 带有 50+ 子路径导出的 Plugin SDK。这是四个项目中唯一真正横切的插件系统。单个插件的 `onActivate()` 方法可以调用：
- `api.registerTool()` -- 添加工具
- `api.registerChannel()` -- 添加通信通道（Discord、Slack 等）
- `api.registerContextEngine()` -- 替换上下文压缩策略
- `api.registerModelProvider()` -- 添加模型后端

一个插件，四个注册目标跨越四个不同的架构模块。这就是它成为横切的原因。SDK 作为独立包发布，这样第三方开发者可以基于稳定的 API 构建，而不依赖 openclaw 的内部实现。

**NemoClaw：** CLI / Plugin / Blueprint -- 三个松散耦合的部分。这里的 "Plugin" 指的是 NemoClaw 自己用于扩展管理 CLI 的插件系统，而非用于扩展 Agent 运行时。在架构意义上不是横切的。

**Claude Code：** 没有 Plugin 框架。工具是静态导入的。特性标志通过死代码消除在构建时控制哪些工具可用。扩展通过修改源码实现，而非通过加载插件。

### 热配置重载

**hermes-agent：** 无热重载。配置在启动时读取。变更需要重启。

**openclaw：** 配置漂移检测。openclaw 定期将运行中的配置与磁盘上的配置文件进行比较。当检测到差异时，受影响的模块被通知重新加载其状态。这是四个项目中唯一的热配置实现。

**NemoClaw：** Agent（沙箱内的 openclaw 实例）没有热重载。NemoClaw 自己的 Blueprint 可以重新应用配置变更，但这涉及重新部署沙箱 -- 不是实时重载。

**Claude Code：** 无热重载。单用户 CLI 工具。重启很简单。

## 对比表

| 维度 | hermes | openclaw | NemoClaw | Claude Code |
|-----------|--------|----------|----------|-------------|
| 日志 | Python `logging`，纯文本 | Sentry + 结构化事件 | 健康状态分类 | 仅内部遥测 |
| 指标 | 无 | Sentry 性能 | 网关探测结果 | 无（面向用户的） |
| 追踪 | 无 | Sentry transactions | 无 | 无 |
| Plugin 框架 | 无（仅工具注册表） | Plugin SDK，50+ 子路径导出，4 个注册目标 | 管理 CLI 插件 | 无 |
| 热配置重载 | 无 | 配置漂移检测 | Blueprint 重新应用（非实时） | 无 |
| 第三方可扩展性 | Fork 仓库 | 发布插件包 | 不适用 | Fork 仓库 |

## 最佳实践

**从第一天就添加结构化日志。** 这是投入最低、回报最高的单一投资。不是 `console.log("tool failed")`。是带有时间戳、级别、模块、操作、持续时间和相关 ID 的结构化 JSON。

```json
{
  "ts": "2024-01-15T10:23:45Z",
  "level": "error",
  "module": "tools",
  "op": "execute",
  "tool": "shell_exec",
  "duration_ms": 14200,
  "error": "timeout",
  "session_id": "abc-123",
  "iteration": 7
}
```

你可以 grep 它。你可以管道到任何日志聚合器。你之后可以构建仪表板。第一天添加它的成本几乎为零。在 6 个月 `console.log` 之后再改造的成本是巨大的。

**在需要第三方可扩展性之前不要构建插件系统。** hermes 有 50+ 工具，没有插件系统。它能用。Claude Code 有 60+ 工具，没有插件系统。它能用。构建 Plugin 框架的触发条件是有一个具体的第三方想要在不 fork 你代码的情况下扩展你的 Agent。如果这个第三方不存在，一个简单的工具注册表（模块 3）就足够了。

**在有多用户部署之前不要构建热配置重载。** 如果重启你的 Agent 需要 2 秒且只影响一个用户（你自己），热重载就是纯开销。openclaw 构建了配置漂移检测，因为它管理的是跨多个通道服务多个用户的持久化 Agent 实例。重启的代价是所有连接用户的断开连接和丢失的会话状态。这就是门槛。

**如果你要构建插件系统，就让它横切，否则不如不做。** 一个只注册工具的插件系统不过是过度工程化的工具注册表。openclaw 的 Plugin SDK 的价值在于，一次插件激活同时触及工具、通道、上下文引擎和模型提供商。如果你的"插件"只添加工具，省掉抽象层直接用 `registry.register()` 吧。

## 逐步构建：如何实现

### 步骤 1：添加结构化日志

选择一个结构化日志库（Python 用 `structlog`，Node.js 用 `pino`）。每条日志行都是一个 JSON 对象，包含时间戳、级别、模块、操作、持续时间和相关 ID。约定：用 `module.operation.phase` 作为事件名称。在整个代码库中保持一致。可搜索。

### 步骤 2：添加基础指标

追踪四个数字：
1. **工具执行次数** -- 按工具名称和成功/失败
2. **模型调用延迟** -- p50、p95、p99
3. **每请求迭代次数** -- Agent 完成任务前循环了多少次
4. **错误率** -- 按模块和错误类型

将这些存储在内存中，通过 `/metrics` 端点暴露（Prometheus 格式）或定期记录日志。暂时不要添加指标数据库 -- 那是你现阶段不需要的基础设施。

### 步骤 3：添加请求追踪（需要时）

当你的 Agent 处理并发请求或委派给子 Agent 时，你需要 trace ID 来关联相关的日志行。在请求开始时生成 UUID，绑定到你的 logger，该请求中后续的每条日志行都包含这个 trace ID。OpenTelemetry 是跨服务分布式追踪的标准。但一个简单的 trace ID 在你的调用链中传播就能覆盖 90% 的调试需求。

### 步骤 4（可选）：添加插件系统

仅当第三方需要扩展你的 Agent 时。

定义一个 `PluginAPI` 接口，为每个模块提供注册方法：`registerTool()`、`registerChannel()`、`registerContextEngine()`、`registerModelProvider()`。一个 `Plugin` 有 `name`、`version`、`onActivate(api)` 和 `onDeactivate()`。关键设计决策：`PluginAPI` 接口决定了你的插件系统的横切程度。如果它只有 `registerTool()`，它就是带额外步骤的工具注册表。openclaw 的 4 目标注册是参考。

### 步骤 5（可选）：添加热配置重载

仅用于重启有用户可感知成本的多用户部署。

模式：一个 `ConfigWatcher` 每 N 秒对配置文件进行哈希。当哈希变化时，它加载新配置并通知订阅者。每个读取配置的模块注册一个回调。Model 服务可能切换提供商。Tools 可能更新速率限制。Context Manager 可能调整压缩阈值。间隔（通常 30 秒）平衡了响应性与文件 I/O 开销。

## 常见陷阱

**1. 把所有东西都叫"横切"。** 命令审批是 Tools。记忆注入是 Context Manager。认证是安全信封。"横切"这个词被用来为把代码放在共享位置辩护，而实际上它属于一个特定模块。问自己："这个关注点是否需要同时修改 3 个以上的模块才能实现？"如果不是，它就不是横切的。

**2. 非结构化日志。** `console.log("Error in tool: " + toolName)` 在规模化时毫无用处。你无法 grep 所有工具错误。你无法计算错误率。你无法构建告警。结构化日志每条语句多花 5 分钟，之后节省数小时的调试。

**3. 在工具注册表稳定之前构建插件系统。** openclaw 的 Plugin SDK 导出 50+ 子路径。这意味着 50+ 个第三方代码依赖的接口。如果这些接口变了，每个插件都会坏掉。先构建和稳定你的工具注册表、上下文引擎和模型服务接口。插件系统是建立在已经稳定的抽象之上的契约层。

**4. 没有版本控制的热重载。** 你在会话中途将模型从 GPT-4 切换到 Claude。现有的会话是以 GPT-4 的系统提示格式开始的。Claude 期望不同的格式。会话崩溃。热重载需要按会话固定配置：新会话获取新配置，现有会话保持旧配置直到结束。

**5. 可观测性作为事后补救。** "我们以后再加日志"意味着你会用 `print()` 语句调试生产问题 6 个月。结构化日志是这个模块中唯一一个没有任何理由推迟的东西。

## 跨模块契约

### 横切关注点对其他模块的期望

| 模块 | 契约 | 原因 |
|--------|----------|-----|
| **所有模块** | 发射结构化日志事件 | 可观测性收集和路由它们 |
| **所有模块** | 暴露配置重载回调（如果实现了热重载） | 配置监听器在变更时通知订阅者 |
| **Tools、Model、Context、Channels** | 提供注册接口 | Plugin 框架将它们包装成统一的 PluginAPI |

### 横切关注点为其他模块提供什么

| 消费方 | 提供内容 | 格式 |
|----------|-----------------|--------|
| **运维人员** | 聚合的日志、指标、追踪 | JSON 日志流、Prometheus 指标、OpenTelemetry spans |
| **基础设施** | 从可观测性派生的健康信号 | 基于错误率或延迟阈值的告警触发器 |
| **第三方开发者** | 稳定的扩展 API | 带版本化注册方法的 PluginAPI 接口 |
| **所有模块** | 更新后的配置值 | 带新配置对象的配置重载通知 |

### 不变量

- 可观测性永远不阻塞 Agent 的主循环。日志是发后即忘的。指标在内存中累积并异步刷新。
- 插件系统不创建新接口。它将已有的注册方法（来自 Tools、Model、Context、Channels）包装成单一激活点。如果一个模块没有注册接口，插件系统就无法扩展它。
- 热配置重载是最终一致的，不是即时的。模块在其下一个自然边界处理配置变更（下一个请求、下一个健康检查周期），而非在操作中途。
- 没有模块依赖于插件系统。插件依赖于模块。如果你移除插件系统，每个模块继续正常工作。这是检验你的插件抽象是否正确分层的测试。
