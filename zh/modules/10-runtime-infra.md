# 模块 10：运行时 + 基础设施
> 你的 Agent 在**哪里**运行（运行时）以及**谁**来设置它（基础设施）。两个层看起来像一个，直到出问题才发现不同。

## 它是什么

运行时是你的 Agent 执行所在的环境。它在会话的整个生命周期内存在。本地 Node.js 进程是运行时。Docker 容器是运行时。带有 Landlock 文件系统隔离、seccomp 系统调用过滤和 L7 网络代理的 OpenShell 沙箱也是运行时。

基础设施是创建、配置和拆除该运行时的机制。它间歇性运行 -- 在初始部署时、配置变更时、出现故障时。把它想象成你的 Agent 的 Terraform：它完成工作然后退出。运行时继续运行。

这两层之间的混淆是普遍的。Docker 横跨两者：`docker create` 是基础设施操作；提供进程隔离的运行容器是运行时。OpenShell 横跨两者：`sandbox create` 是基础设施；在运行时过滤网络流量的 L7 代理是运行时。保持这种区分清晰可以防止你把部署逻辑混入 Agent 的热路径中。

大多数 Agent 以本地进程开始，永远不需要更多。运行时 + 基础设施在你需要隔离（不受信任的代码执行）、可复现性（跨机器的一致环境）或多租户部署（每个用户有自己的沙箱）时变得重要。

## 它解决什么问题

**隔离。** 你的 Agent 执行 shell 命令。如果这些命令可以不受限制地访问主机文件系统、网络和进程，一次错误的工具调用就可能危及整台机器。运行时隔离限制了爆炸半径。

**可复现性。** "在我的机器上能跑"这句话会毁掉 Agent 部署。容器化运行时保证每次都是相同的 Python 版本、相同的系统库、相同的文件布局。

**生命周期管理。** Agent 需要配置推理端点、应用安全策略、运行健康检查。没有结构化的基础设施层，这些就变成了没人维护、所有人都忘记的临时脚本。

**恢复。** 容器崩溃。沙箱触及资源限制。进程挂起。基础设施处理检测和恢复 -- 重启容器、重建沙箱、故障转移到备份。没有它，你的 Agent 会静默死亡，直到用户投诉才会有人注意到。

## 四个项目如何实现

### Claude Code

**运行时：** 本地进程。你运行 `npm install -g @anthropic-ai/claude-code`，然后 `claude`。它在你的机器上启动一个 Node.js 进程。没有容器。没有沙箱。除了操作系统级别的用户权限外没有隔离。

**基础设施：** 无。npm 包管理器是最接近基础设施层的东西。没有健康检查、没有自动恢复、没有部署编排。

**为什么可行：** Claude Code 是单用户开发者工具。用户就是运维人员。如果进程挂了，用户重启它。如果工具调用做了破坏性操作，权限系统在执行前拦截。权衡是显式的：零运维开销换取零隔离。

### hermes-agent

**运行时：** 默认为本地进程。部署到服务器时为 Docker 容器。还支持 Modal（云函数）和 SSH（远程执行）作为替代后端。

**基础设施：** 用于容器部署的 Dockerfile 和 docker-compose。用于无服务器的 Modal 配置。用于远程后端的 SSH 连接管理。每个后端是独立的代码路径 -- 没有统一的基础设施抽象。

**多后端支持：** hermes 是唯一一个支持在部署时选择运行时的项目。本地用于开发，Docker 用于预发布，Modal 用于突发工作负载，SSH 用于现有服务器。灵活性的代价是维护四条独立的部署路径。

### openclaw

**运行时：** Agent 进程使用 Docker 容器。云部署使用 Fly.io。组件间安全联网使用 Tailscale。

**基础设施：** Docker Compose 编排。Fly.io 部署配置。没有类似 Terraform 的抽象 -- 部署是命令式脚本和特定于平台的配置文件。

**网络：** Tailscale 值得关注。它在 Agent、其工具和外部服务之间创建私有网状网络。这解决了"我的容器化 Agent 如何访问我的本地数据库"的问题，而无需向互联网暴露端口。

### NemoClaw

**运行时：** OpenShell 沙箱。这是四个项目中最成熟的运行时。每个沙箱获得：
- **Landlock** 文件系统隔离 -- 配置文件只读，可写数据限制在特定路径
- **seccomp** 系统调用过滤 -- 限制进程可以调用哪些系统调用，以非 root 运行，移除危险二进制文件（gcc、netcat）
- **L7 网络代理** -- 按二进制文件的网络策略。不仅是"Agent 可以访问哪些域"，而是"哪个特定二进制文件可以访问哪个域"。如果 curl 不在白名单中，即使 Agent 有 shell 访问权限，通过 curl 的数据泄露也不可能
- **SSRF 防护** -- DNS 解析后进行 IP 验证，阻止内部网络访问

**基础设施：** Blueprint 模型。这是 NemoClaw 最突出的贡献。

Blueprint 的工作方式类似 Terraform：`plan`（计算需要更改什么）、`apply`（执行更改）、`status`（验证当前状态）、`rollback`（回滚到之前的状态）。上线流程是一个 7 步向导，支持检查点恢复 -- 如果第 4 步失败，你从第 4 步重新开始，而非第 1 步。

关键架构洞察：**NemoClaw 本身是纯基础设施。** 它编排 OpenShell 命令、配置 Docker 容器、设置推理端点。部署完成后，NemoClaw 退出。运行时（OpenShell 沙箱 + L7 代理 + 容器内的 openclaw 实例）在没有 NemoClaw 的情况下继续运行。

**双层组件：**
```
NemoClaw  -> 纯基础设施（编排完就退出）
OpenShell -> 两层都有（sandbox create = 基础设施，L7 proxy = 运行时）
Docker    -> 两层都有（docker create = 基础设施，容器隔离 = 运行时）
Linux     -> 纯运行时基础（Landlock, seccomp, cgroups, namespaces）
```

## 对比表

| 维度 | hermes | openclaw | NemoClaw | Claude Code |
|-----------|--------|----------|----------|-------------|
| 默认运行时 | 本地进程 | Docker 容器 | OpenShell 沙箱 | 本地进程 |
| 隔离级别 | 无（本地）/ 基础（Docker） | 基础（Docker） | 强（Landlock + seccomp + L7） | 无 |
| 替代运行时 | Docker、Modal、SSH | Fly.io | Docker（作为内层） | 无 |
| 基础设施模型 | 按后端的脚本 | Docker Compose + 平台配置 | Blueprint（plan/apply/status/rollback） | npm install |
| 健康检查 | 无 | 无 | 有（gateway-state.ts 健康分类） | 无 |
| 自动恢复 | 无 | 无 | 有（检测死亡 -> SSH 探测 -> 重启 -> 注册表恢复） | 无 |

| 网络 | 直连 | Tailscale 网状网络 | 按二进制文件的 L7 代理 | 直连 |

## 最佳实践

**从本地进程开始。** `npm install` 或 `pip install`，然后直接运行。Claude Code 以这种方式服务数百万用户。在你有具体理由之前，不要添加 Docker、沙箱或云函数。

**部署到服务器时添加 Docker。** 当你的 Agent 在不是你笔记本电脑的机器上运行时，你需要可复现的环境。一个固定语言版本、安装依赖、复制代码的 Dockerfile 是最低要求。hermes 的方式 -- Dockerfile 与源码放在一起 -- 就是模板。

**仅在执行不受信任代码时添加沙箱隔离。** 如果你的 Agent 运行任意用户提供的代码（不只是预定义的工具），你需要沙箱级别的隔离。NemoClaw 的方式 -- Landlock 管文件系统，seccomp 管系统调用，L7 代理管网络 -- 是黄金标准。但它需要 Linux，增加运维复杂性，如果你的 Agent 只调用定义好的工具则是过度设计。

**在代码中将基础设施与运行时分离。** 即使你从 `docker-compose up` 开始，也要将"创建和配置环境"的代码与"Agent 在此环境中运行"的代码分开。NemoClaw 的 Blueprint 模型是理想选择。至少，让 `deploy.sh` 不是 `start.sh`。

**健康检查是廉价的保险。** 定期对你的 Agent 健康端点进行 HTTP ping 可以在用户发现之前捕获崩溃。NemoClaw 的 `gateway-state.ts` 将健康状态分为不同类别（健康、降级、死亡）并触发相应响应。你可以从简单的存活检查开始，之后再增加复杂性。

## 逐步构建：如何实现

### 步骤 1：从本地开始

`python agent.py` 或 `node agent.js`。不要 Docker。不要容器。Claude Code 就停在这里，对于单用户工具来说这是一个有效的停止点。

### 步骤 2：添加 Dockerfile

部署到服务器时：固定语言版本，`COPY` 源码，暴露端口。将 Dockerfile 放在仓库根目录。如果你的 Agent 需要旁边有数据库，添加 `docker-compose.yml`。

### 步骤 3：将部署与运行分离

创建两个入口点：`deploy.sh`（基础设施 -- 构建镜像、创建容器、配置端点）和 `start.sh`（运行时 -- 启动 Agent 进程）。部署脚本运行一次。启动脚本持续运行。不要合并它们。

### 步骤 4：添加健康检查

一个返回 `{"status": "healthy", "uptime_seconds": N}` 的 `/health` 端点。基础设施轮询它。连续 N 次非 200 响应触发恢复（重启进程、重建容器）。

### 步骤 5（可选）：添加 Blueprint 风格的基础设施

仅当你管理多个 Agent 实例或需要可复现的多步骤部署时。

接口：`plan(desired_state) -> steps`、`apply(steps) -> result`、`status() -> current_state`、`rollback(checkpoint) -> result`。四个方法，像 Terraform 一样。

NemoClaw 的 7 步上线向导是参考。每一步（创建沙箱、配置网络、部署 Agent、设置推理、应用安全策略、验证健康、注册到网关）成功后设置检查点。第 5 步失败意味着从第 5 步恢复，而非第 1 步。

### 步骤 6（可选）：添加沙箱隔离

仅当你的 Agent 执行不受信任的代码时：

```bash
# 文件系统隔离（仅 Linux）
landlock --ro /etc --ro /usr --rw /app/data

# 系统调用过滤
seccomp --default deny --allow read,write,open,close,mmap,...

# 网络隔离
l7-proxy --allow "node:api.openai.com:443" --deny-all
```

这是 NemoClaw 的领域。按二进制文件的网络策略是关键创新：即使攻击者在沙箱内获得了 shell 访问权限，他们也无法泄露数据，因为只有 Agent 二进制文件（不是 curl、不是 wget、不是 netcat）被允许发起网络请求。

## 常见陷阱

**1. 过早容器化。** 你的 Agent 有 3 个用户，运行在你的笔记本电脑上。你花了两天调试 Docker 网络问题而不是改进你的 Agent。在你真正需要隔离或多机器部署之前，本地进程就够了。

**2. 基础设施在热路径中。** 你的 Agent 在每次工具调用时检查"我的容器健康吗？"。健康检查应该在单独的进程或定时任务中，而不是在请求路径中。NemoClaw 的 `gateway-state.ts` 独立于 Agent 的主循环运行。

**3. 没有检查点恢复。** 你的 7 步部署在第 6 步失败。你从第 1 步重新运行，需要 10 分钟。NemoClaw 用基于检查点的恢复解决了这个问题。如果你有多步骤部署，每步之后设置检查点。

**4. 混淆作为基础设施的 Docker 和作为运行时的 Docker。** `docker build` 和 `docker create` 是基础设施操作。运行容器提供的进程隔离和资源限制是运行时。当人们说"用 Docker 就行了"时，他们通常指两层但没有意识到。保持你的心智模型清晰。

**5. 没有威胁模型的沙箱。** 你添加了 Landlock、seccomp 和 L7 代理因为"安全"。但你的 Agent 只调用天气 API 和读取本地文件。沙箱管理的运维开销超过了安全收益。在添加隔离层之前，写下你要防范什么。

## 跨模块契约

### 运行时 + 基础设施对其他模块的期望

| 模块 | 契约 | 原因 |
|--------|----------|-----|
| **Agent Core** | 健康端点或心跳信号 | 基础设施需要知道 Agent 是否存活 |
| **Model 服务** | 推理端点配置 | 基础设施配置 Agent 连接哪个模型端点 |
| **安全信封** | 安全策略定义 | 基础设施在部署时应用文件系统、网络和系统调用策略 |
| **Tools** | 资源需求声明 | 基础设施需要知道工具需要什么系统资源（网络访问、文件系统路径、GPU） |

### 运行时 + 基础设施为其他模块提供什么

| 消费方 | 提供内容 | 格式 |
|----------|-----------------|--------|
| **Agent Core** | 隔离的执行环境 | 配置了权限的进程/容器/沙箱 |
| **Model 服务** | 配置好的推理连接 | 到模型 API 的网络路由，凭据作为环境变量注入或挂载为密钥 |
| **Tools** | 文件系统和网络边界 | 可写路径、允许的网络目标、可用的系统二进制文件 |
| **安全信封** | 安全策略的执行 | Landlock 规则、seccomp 过滤器、L7 代理规则 -- 在操作系统层面执行，而非应用层面 |

### 不变量

- 基础设施操作永远不在 Agent 的主循环内运行。它们在 Agent 启动之前运行，或在单独的管理进程中运行。
- 运行时隔离对 Agent 是透明的。Agent 不知道（也不应该关心）它是在本地进程中运行还是在 OpenShell 沙箱中运行。相同的代码在两种环境中运行。
- 健康检查是拉取式的（基础设施轮询 Agent），而非推送式的（Agent 向基础设施报告）。这样，一个崩溃的、无法推送的 Agent 仍然会被检测到。
- 检查点状态存储在它所管理的运行时之外。如果沙箱崩溃，检查点数据得以保存用于恢复。
